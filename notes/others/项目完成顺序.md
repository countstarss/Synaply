这个顺序是对的，而且我会建议你们就按这个节奏走。

现在先不新增 `Docs` 模块实现，先把 `Project` 和 `Workflow` 两条重构线分别收口，再做一次集成整理，最后再落 `Docs`，这是更稳的路径。原因很简单：

1. `Docs` 天然依赖 `Project` 的上下文容器能力。
2. `Docs` 也依赖 `Workflow` 的执行语义和输入/输出关系。
3. 如果上游两个模块还在变，`Docs` 现在开工，后面大概率要反复返工。
4. 你们现在最需要控制的是复杂度，不是同时铺更多模块。

我建议接下来按这个方式由我来统筹：

1. 先冻结 `Docs` 实现层，只保留 brief，不开正式开发。
2. `Project` 重构先落地，但重点看它有没有把“项目上下文层”补起来，而不是只改页面。
3. `Workflow` 重构随后落地，重点看它有没有把“执行语义层”补起来，而不是只优化模板页。
4. 两边都完成后，我做一次统一的 integration review，专门找冲突点和断点。
5. 只有当 `Project + Workflow` 的共享契约稳定后，再启动 `Docs`。

我会重点盯这几个冲突区：

1. `projectId / workflowId / issueId` 的关系边界是否稳定。
2. `visibility / owner / assignee / context` 这些共享字段是否统一。
3. 项目页、工作流页、issue 详情里的上下文表达有没有打架。
4. activity / event contract 是否已经足够稳定，能支撑后续 docs 接入。
5. 页面结构是不是已经为 `Docs` 预留了自然入口，而不是以后再硬塞。

`Docs` 适合启动的时机，我会用这几个标准来判断：

1. `Project` 已经能稳定承载 `brief / status / risks / linked resources` 这一层上下文。
2. `Workflow` 已经能稳定表达运行态、交接态和产出态。
3. `Project` 和 `Workflow` 的 API/模型已经不再频繁改字段。
4. 我们已经清楚 `Docs` 应该挂在哪些页面、以什么关系接入。

所以结论很明确：  
现在不急着做 `Docs` 实现，是对的；等 `Project` 和 `Workflow` 逐个解决完，我来做统一收口，再启动 `Docs`，复杂度会低很多，返工也会少很多。
