基于当前实现，我的判断是：你们现在不是“没有这些模块”，而是这些模块还没有形成一条单向推进的工作链。更准确地说，Project / Issue / Workflow / Doc / Inbox 目前更像 5 个并排页面，而不是 1 条共享上下文里的交付路径。

## 主要卡点

1. Inbox 的定义跑偏了。现在的 /inbox 实际挂的是邮件客户端 CachedInboxPage.tsx (line 3)，数据来自本地 mockEmails use-mail-store.ts (line 48) 和 mock-data.ts (line 38)，而且导航里它仍被标成 isReady: false constant.ts (line 31)。这说明主链最后一环还不是“协作动作中心”，而是一个独立 demo。

2. Docs 还是平行系统，没有回到项目上下文。文档页虽然技术上支持按 `projectId` 过滤，但当前团队文档页还没有把项目上下文真正接回主链。结果是 docs 存在，但不是 project 的背景、决策和方案层，只是另一个入口。

3. Project 现在更像 issue 容器，不像协作中枢。Project 模型本身只有基础字段和 issues 关系 schema.prisma (line 88)，项目详情组件暴露出来的核心也几乎都围绕 projectIssues、issue 看板和创建 issue ProjectDetailView.tsx (line 37)。这和你们想表达的“为什么做、先做什么、下一步做什么”还有距离。

4. Workflow 还是模板系统，不是可靠的执行系统。工作流页的文案和统计都围绕“模板管理”展开 WorkflowsPageContent.tsx (line 240)，统计的也是模板数量 useWorkflowApi.ts (line 274)。更关键的是，工作流 Issue 的创建不是原子性的：后端 DTO 只定义了工作流字段 create-workflow-issue.dto.ts (line 13)，服务端也只先落一个最小 Issue issue.service.ts (line 208)，前端再二次 patch 项目、状态、优先级等 useIssueApi.ts (line 32) useIssueApi.ts (line 220)。这会让“workflow-driven execution”看起来成立，但底层并不稳。

5. Sync 现在是技术层广播，不是产品层节奏。实时事件目前主要覆盖 issue、comment 和 workflow node 变化 events.ts (line 1)，但没有一个统一的“谁该处理什么、哪里卡住了、哪些决定变了”的产品视图。所以你们有实时性，但还没有“同一份进度感”。

## 最需要重点克服的地方

1. 先把 Inbox 重新定义成行动层，而不是消息层。它应该承接 review request、handoff、blocker、mention、decision update、workflow 到我这里了，而不是邮件列表。

2. 把 Project 做成上下文容器。项目页至少要容纳 Brief / Linked Docs / Active Workflows / Issues / Risks / Pending Reviews，否则项目不会成为团队的默认落点。

3. 把工作流执行改成服务端一次性建模。哪怕暂时不引入 WorkflowRun，也要先保证 workflow issue 在一次请求里完成 issueType / project / state / priority / visibility / assignee 的正确落库。

4. 把文档正式接回主链。最低要求是项目级文档必须能自然进入项目页，项目删除/归档也要处理相关 docs 生命周期；之后再考虑 issue 级文档。

5. 再做 Personal Pulse / Team Pulse。这是主链打通后的首页形态，不是单独再加一个 dashboard。

## 预期应该达到的效果
理想状态下，团队进入一个项目时，不需要再分别去翻项目页、issue 页、workflow 页、docs 页。一个项目页就能回答：这个项目为什么做、现在推进到哪一步、卡在哪、谁要接手、相关方案和决策在哪里。

再往前一步，任何一个动作都应该自然带着上下文流动：从项目里创建 issue，会自动带上项目语境；issue 进入 workflow，会形成明确交接；相关 doc 的更新或关键 decision，会自动进入 inbox；每个人首页看到的是“我现在该处理什么”，而不是“我需要去哪些模块里自己拼信息”。
