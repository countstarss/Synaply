# Synaply 产品原则

## 产品定位

Synaply 是为小型初创团队打造的远程协作软件。

它面向 3 到 15 人的小型初创团队，适合产品、设计、研发、运营等多角色混合协作场景。产品承诺不是“管理更多任务”，而是“帮助团队更快、更清晰地完成远程协作，并把工作推进到交付”。

Synaply 的核心产品结构应当是：

- `Projects` 用来定义范围、目标和方向。
- `Issues` 用来承载可执行的工作单元。
- `Workflows` 用来让推进路径与交接过程可见。
- `Docs` 用来沉淀背景、决策和共享理解。
- `Inbox / Sync` 用来同步变化、提醒重点、维持节奏感。

Synaply 不应演变成一个泛化的项目管理套件。它应当始终保持高端、克制、专业的协作软件定位，围绕结构化执行与协作 momentum 展开。

## 功能判断标准

评估任何一个新功能时，都应先问这三个问题：

1. 它有没有让跨角色交接更清晰？
2. 它有没有减少远程协作中反复催促、开会或追状态的需要？
3. 它有没有增强 `Projects / Issues / Workflows / Docs` 之间的一体化上下文？

如果一个功能只满足其中一条，就应当谨慎推进。

## 必要功能

这些能力是支撑 Synaply 定位成立的必要部分。

### 1. 交接与评审机制

产品应支持明确的角色交接动作，例如：

- 请求产品确认
- 请求设计评审
- 请求研发接手
- 请求运营跟进发布

理由：
Synaply 的核心是多角色协作，而不是个人任务追踪。没有明确交接，`Workflow` 就只是状态装饰，而不是真正的协作流转。

### 2. 阻塞与依赖

每个 `Issue` 都应能清楚表达：

- 是否被阻塞
- 被什么阻塞
- 谁负责解除阻塞
- 预计何时恢复推进

理由：
远程团队里的很多风险不是任务不够多，而是等待不可见、卡点不透明。阻塞必须能够被看见，而不是靠人去追问。

### 3. 决策记录

产品应当记录关键决策，并与对应的 `Project`、`Issue`、`Doc` 保持关联。

理由：
如果 `Docs` 不只是归档结果，而是协作过程的一部分，那么产品不仅要记录“做了什么”，还要记录“为什么这样做”。

### 4. Personal Pulse 与 Team Pulse

产品应当清楚回答两个问题：

- 我现在最该处理什么？
- 团队当前的风险点在哪里？

理由：
协作软件的价值，不是展示更多信息，而是降低下一步的不确定性。首页应优先回答优先级与推进状态，而不是堆叠数据面板。

### 5. 异步摘要

提供简明的日摘要或周摘要，覆盖：

- 项目推进情况
- 工作流状态变化
- 当前风险
- 待确认事项

理由：
remote-first 团队需要共同节奏，但这种节奏应建立在结构化同步之上，而不是通知轰炸之上。

### 6. 模板系统

提供轻量但实用的协作模板，例如：

- 项目 Brief
- 需求文档
- 设计评审
- 发布清单
- 复盘模板

理由：
小团队真正需要的是一致性，而不是过度配置。模板能减少重复发明流程的成本，也能强化良好的协作习惯。

### 7. 轻量集成

优先集成：

- `GitHub`，承接研发执行
- `Slack`，承接通知分发与协作信号

理由：
这些集成有助于团队采用 Synaply，但不应让产品本身滑向聊天工具或开发者专属工具。集成的目标是桥接流程，而不是成为新的中心。

## 值得做，但不应优先做

这些方向与产品定位一致，但应建立在核心协作链路已经扎实的前提下。

### 1. 里程碑与发布视图

适合把多个 `Issue` 收束成阶段性交付，但优先级低于交接、阻塞和决策清晰度。

### 2. 协作节奏工具

例如周计划、异步 standup、复盘等。

这些能力有助于强化团队节奏，但前提是底层对象关系和工作流流转已经稳定。

### 3. 角色视图

不同角色可以拥有不同入口与视角，但底层系统应保持统一。

它能提升可用性，但不应过早演变成复杂权限系统或高度可配置仪表盘工程。

### 4. AI Assist

适合的方向包括：

- 自动总结项目变化
- 提取下一步行动
- 识别潜在风险

AI 的角色应当是放大清晰度，而不是替代结构本身。只有在系统结构清楚之后，AI 才能真正产生价值。

## 当前不建议做的方向

这些方向大概率会稀释 Synaply 的产品焦点。

### 1. 内置聊天

这会把产品重心拉向消息工具，削弱其在结构化协作上的独特价值。

### 2. 重型计划工具

避免过早投入复杂甘特图、资源排期、工时记录、企业式容量管理等能力。

这些能力更适合大型组织和重流程场景，不适合当前定位。

### 3. 大而全的管理套件

避免扩展为完整的 OKR、CRM 或泛化知识门户。

产品覆盖面越大，Synaply “统一协作中枢”的承诺就越容易变弱。

### 4. 复杂审批流引擎

Synaply 应当追求“轻量但严谨”，而不是演变成企业级流程审批系统。

## 推荐的产品推进顺序

产品 roadmap 建议按以下顺序推进：

1. 先彻底打通 `Project -> Issue -> Workflow -> Doc -> Inbox` 这条核心链路。
2. 再补齐关键协作动作：交接、阻塞、决策、摘要。
3. 最后补充强化层：模板、集成、AI Assist、角色视图。

简而言之：

不要在协作尚未自然流转之前，继续扩张更多模块。

Synaply 应当靠“让工作更容易被推进”取胜，而不是靠“拥有更多功能”取胜。
